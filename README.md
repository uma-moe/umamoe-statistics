# Uma.moe Statistics Generator

Standalone Rust port of `scripts/statistic.py` for weekly Team Stadium statistics builds.

The generator is designed for the big table case: it streams rows from PostgreSQL with `COPY`, keeps only aggregate counters in memory, and writes the same frontend-facing layout:

```text
statistics/
├── datasets.json
└── {dataset}/
    ├── index.json
    ├── global/global.json
    ├── distance/{1,2,3,4,5}.json
    └── characters/{character_id}.json
```

Generated datasets use the ID-only format marked by `format: "ids-v1"` and `format_version: 2` in `datasets.json`, each dataset `index.json`, and per-file metadata. Older datasets without this marker are legacy name-resolved statistics.

## Run

From the repository root:

```dotenv
DATABASE_URL=postgresql://user:password@127.0.0.1:5432/honsemoe_db
```

```powershell
cargo run --release
```

The CLI loads `.env` automatically for local runs. You can still pass `--database-url` or set `DATABASE_URL` in the shell if you prefer.

Useful options:

```powershell
cargo run --release -- `
  --dataset-version 2026-W19
```

For a small smoke test:

```powershell
cargo run --release -- --limit 100000
```

To watch live exporter load while a full export runs:

```powershell
cargo run --release -- --resource-usage
```

Support-card counts, skill counts, and exact support-card deck combinations are generated by default. Deck combinations are stored with compact in-memory keys and are converted to frontend JSON keys only when output files are written.

## Docker

Build the image:

```bash
docker build -t umamoe-statistics-generator .
```

Run locally and write JSON to `./statistics`:

```bash
docker run --rm \
  --env DATABASE_URL="postgresql://user:password@host.docker.internal:5432/honsemoe_db" \
  --add-host host.docker.internal:host-gateway \
  -e UMAMOE_OUTPUT_DIR=/output/statistics \
  -v "$PWD/statistics:/output/statistics" \
  umamoe-statistics-generator
```

Run on the server and publish into both frontend roots:

```bash
docker run --rm \
  --env-file /opt/umamoe-statistics/statistics.env \
  --add-host host.docker.internal:host-gateway \
  -e UMAMOE_TARGET_ROOTS="/var/www/html/umamoe /var/www/html/umamoe_beta" \
  -v /var/www/html/umamoe:/var/www/html/umamoe \
  -v /var/www/html/umamoe_beta:/var/www/html/umamoe_beta \
  umamoe-statistics-generator
```

The Docker wrapper uses these environment values:

| Variable | Required | Purpose |
| --- | --- | --- |
| `DATABASE_URL` | yes | PostgreSQL connection string used by the generator. |
| `UMAMOE_TARGET_ROOTS` | deploy yes | Space-separated frontend roots to publish into, for example `/var/www/html/umamoe /var/www/html/umamoe_beta`. |
| `UMAMOE_STATISTICS_RELATIVE_DIR` | no | Statistics directory below each target root. Defaults to `assets/statistics`. |
| `UMAMOE_OUTPUT_DIR` | no | Output directory when `UMAMOE_TARGET_ROOTS` is not set. Defaults to `/output/statistics`. |
| `UMAMOE_DATASET_VERSION` | no | Dataset id such as `2026-W19`. If omitted, the generator uses the current date. |
| `UMAMOE_LIMIT` | no | Optional row limit for smoke runs. |
| `UMAMOE_PROGRESS_EVERY` | no | Progress logging interval. Defaults to `250000`. |
| `UMAMOE_RESOURCE_USAGE` | no | Set to `true` or `1` to print exporter, Postgres, and system CPU/memory at progress checkpoints. |

When publishing, each dataset is written to a hidden staging directory first, then moved into place. `datasets.json` is updated after the dataset files exist, so the frontend should not observe a new dataset before its files are ready.

## GitHub Actions Deploy

The deploy workflow mirrors the backend image flow: it builds the Docker image, ships it to the host over SSH, loads it, and runs the generator against mounted frontend directories.

It also runs automatically every Sunday at 00:00 JST, which is Saturday 15:00 UTC for GitHub Actions cron. Scheduled runs use the normal defaults: full dataset, deploy target `both`, and the dataset version generated by the exporter.

Manual runs include a `run_statistics_now` checkbox. Leave it checked to deploy the Docker image and run the export immediately. Uncheck it to only transfer/load the Docker image on the server without generating new statistics.

Repository or environment secrets needed by `.github/workflows/deploy-statistics.yml`:

| Secret | Purpose |
| --- | --- |
| `DEPLOY_HOST` | SSH host for the server. |
| `DEPLOY_PORT` | Optional SSH port. Defaults to `22`. |
| `DEPLOY_SSH_KEY` | Private key for the `github` deploy user. |
| `DEPLOY_KNOWN_HOSTS` | Known-hosts entries for strict host verification. |

Server files expected by the workflow:

| Path | Purpose |
| --- | --- |
| `/opt/umamoe-statistics/statistics.env` | Docker env file. Must include `DATABASE_URL=...`; optional values like `UMAMOE_PROGRESS_EVERY=250000` can live here too. |
| `/var/www/html/umamoe/assets/statistics` | Production statistics target. |
| `/var/www/html/umamoe_beta/assets/statistics` | Beta statistics target. |

## Server Scheduling

The server should provide `DATABASE_URL` through the environment or secret store. Example weekly cron:

```bash
0 3 * * 1 cd /srv/umamoe-statistics && /usr/local/bin/cargo run --release -- --output-dir /var/www/html/umamoe/assets/statistics --dataset-version "$(date +%G-W%V)"
```

For production, build once and schedule the binary instead:

```bash
cargo build --release
DATABASE_URL="postgresql://..." ./target/release/umamoe-statistics-generator --output-dir /var/www/html/umamoe/assets/statistics --dataset-version "$(date +%G-W%V)"
```

## Notes

- The database password is intentionally not hardcoded. Use `DATABASE_URL` or `--database-url`.
- Names, icons, colors, and support-card types are intentionally not resolved by the generator in `ids-v1`; resolve them in the frontend from local data.
- Percentiles are exact for integer stats. The Rust port stores value frequencies instead of all 25M rows.
- The query is ordered by `trainer_id`, which lets trainer-level counts be finalized one trainer at a time without keeping trainer sets in memory.
